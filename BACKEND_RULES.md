# RetailMedia System - バックエンド開発ルール (Backend Rules)

## 🚨 必須ルール1: データ永続化 (S3保存) の徹底

機能追加や既存機能の改善において、バックエンド（`server_retail_dist.js` 等）でデータを変更するエンドポイント（POST, PUT, DELETE）を作成・修正する場合は、**必ずS3への保存関数を呼び出すこと**を必須とします。

メモリ上の変数を更新しただけでは、システム再起動時（AppRunner等のコンテナ再起動）に変更内容がすべて消失し、致命的なデータロストに繋がります。

### 保存関数の使い分け

エンドポイントで扱うデータの種類に応じて、以下のいずれかの関数を必ず同期的に呼び出してください。

#### 1. `saveDatabase()`
- **対象データ**: ユーザー情報 (`users`)、サイネージ設定 (`storeData`, `signageState`)、キャンペーン情報 (`campaigns`)、実績データ (`transactions`) など、システム全般のデータ。
- **使用例**:
  ```javascript
  app.post('/api/example', (req, res) => {
      users[req.body.email] = req.body.data;
      if (typeof saveDatabase === 'function') saveDatabase(); // 必須
      res.json({ success: true });
  });
  ```

#### 2. `saveFinanceDB()`
- **対象データ**: 出金リクエスト (`withdrawalRequests`)、本人確認データ (`kycRequests`)、振込先口座情報 (`creatorBanks`)、代理店データ (`agencyReferrals`) などの金融・審査系データ。
- **使用例**:
  ```javascript
  app.post('/api/creator/withdraw', (req, res) => {
      withdrawalRequests.push(req.body.request);
      if (typeof saveFinanceDB === 'function') saveFinanceDB(); // 必須
      res.json({ success: true });
  });
  ```

## 🚨 必須ルール2: 認証とセキュリティ (Authentication & Security)

ユーザーの認証状態や機密情報を扱う際は、以下のセキュリティ基準を遵守してください。

1. **パスワードの平文保存禁止**: 
   新規登録やパスワードリセット時、パスワードは必ず `crypto.scryptSync` 等を用いてハッシュ化してから保存すること。
2. **グローバル変数によるセッション管理の禁止**:
   `currentUser` のようなグローバル変数にログイン状態を保持すると、複数ユーザーからの同時リクエスト時にセッション情報が混同する（他人のアカウントにログインしてしまう）致命的なバグが発生します。
   **必ず JWT (JSON Web Token) と HTTP-Only Cookie を用いてリクエスト単位で認証状態を管理** してください。

---

## 🚨 必須ルール3: 重要処理の排他制御 (Mutex & Concurrency)

出金処理やポイント付与など、**二重実行されるとシステムや財務に致命的な影響を与えるAPI** を実装する際は、必ずメモリ上の排他制御（Mutex）を組み込んでください。

- **実装例**:
  ```javascript
  const processingTasks = new Set();
  
  app.post('/api/critical/task', (req, res) => {
      const taskId = req.body.taskId;
      if (processingTasks.has(taskId)) {
          return res.status(409).json({ error: "現在処理中です" }); // 連打防止
      }
      processingTasks.add(taskId);
      
      try {
          // ... 重要な処理 ...
      } finally {
          processingTasks.delete(taskId); // 確実なロック解除
      }
  });
  ```

## 🚨 必須ルール4: スケーラビリティとメモリ管理 (Scalability)

現在進行中の SQLite への移行フェーズにおけるルールです。

1. **巨大なインメモリ同期処理の回避**:
   `users` や `financeDB` などの全件データが入った辞書（オブジェクト）に対する同期的な巨大JSON変換は、イベントループをブロックしサーバーを停止させる原因となります。
2. **新規機能のデータベース設計**:
   今後新たに追加するデータモデルや大量のトランザクションが予想される機能（出金履歴、ログデータなど）は、極力JSONではなく **SQLite (`db_sqlite.js`) を用いた非同期処理** で設計してください。

---

## 🚨 必須ルール5: 外部API・LLMの耐障害性とモデルの動的フォールバック (LLM API & Resilience)

Google Geminiなどの外部生成AI APIを使用する際は、モデルの廃止やAPI仕様の変更によってシステムが突然停止（500エラー等）することを防ぐため、以下の設計基準を遵守してください。

1. **複数モデルによる自動フォールバックとリトライ機構の搭載**
   単一 of モデル名（例: `gemini-2.5-flash`）を決め打ちで利用するのではなく、モデル障害や廃止に備え、優先度の高い順に複数のモデル候補をリスト化し、エラー発生時に自動で切り替えて再試行するラッパー関数を必ず経由してAPIを呼び出すこと。
   - **優先度リストの例**: `['gemini-2.5-flash', 'gemini-1.5-flash', 'gemini-2.5-pro']`
2. **通信障害やキー未設定時のフォールバック（デモ動作）の保証**
   APIキーが設定されていない場合や、通信障害等によりすべてのモデルで応答が得られなかった場合でも、システム自体をクラッシュ（500エラー）させず、事前に用意された適切なデモ用（フォールバック）データを返却してフロントエンドの表示や最低限のサービス継続を保証すること。

## 🚨 必須ルール6: 開発中のファイル破損・タグ整合性チェック (File Integrity & Syntax Validation)

セッション切り替え時などのコンテキスト忘却（忘れること）によるバグ再発を防ぐため、HTML/JSファイルを編集またはGitプッシュする際は、以下のファイル整合性チェックを必ず実行してください。

1. **スクリプト・HTMLタグの「数」と「末尾」の整合性検証**
   - 単に正規表現で `<script>`〜`</script>` の間を切り出して文法チェックをかけるだけでは、**「閉じタグ自体が欠落してファイルが途中で切れている破損」** を検知できません。
   - ファイルを編集した際は、必ず `<script>`と`</script>`、`<form>`と`</form>`、`<div>`と`</div>` などの**開きタグと閉じタグの総数が一致していること**、およびファイル末尾が `</html>` などの正規のタグで閉じられているかをプログラム的または目視で必ず検証すること。
2. **Gitプッシュ前の構文チェック実行義務**
   - どのような軽微な変更であっても、Gitプッシュを行う直前には、プロジェクト内にある構文検証スクリプトを実行し、エラー（Syntax Error）が一切存在しないことを確認してからプッシュすること。

## 🚨 必須ルール7: ネイティブモジュールと実行環境の互換性確保および不要な依存の排除

1. **実行環境（GLIBC等）の差異によるネイティブバイナリクラッシュの防止**
   - `sqlite3` や `bcrypt` 等のC/C++ネイティブモジュールを伴うNode.jsパッケージは、本番環境（AWS App Runner等）のOS・GLIBCバージョンと、パッケージが要求するコンパイル済みバイナリの不一致により `ERR_DLOPEN_FAILED (GLIBC_X.XX not found)` を起こすリスクが極めて高い。
   - ネイティブモジュールに依存するパッケージを利用する場合、本番デプロイ時に実行環境のライブラリに合わせて確実にその場でネイティブビルドを強制する設定（例：`package.json` の `prestart` における `npm rebuild --build-from-source` の実行）を必ず仕込んでおくこと。
2. **本番環境とローカル開発環境の依存切り分け・クリーンロードの徹底**
   - 本番環境でPostgreSQLを利用する一方、ローカルでSQLiteを利用するなど、環境ごとにデータベース等の接続先を切り替える設計にする場合、本番で不要なはずのモジュール（例：`sqlite3`）を無条件に `require` させてサーバーを起動クラッシュさせないこと。
   - 使用しないデータベースモジュールは、環境変数などの実行条件に応じて動的（ダイナミック）にインポートするか、不要な依存関係をコードから完全に排除して本番環境の動作をクリーンに保つこと。

## 🚨 必須ルール8: AI向け開発・デプロイ検証ルール（周辺影響 of 徹底検証）

コードの生成、変更、またはデプロイ手順の適用を行う際は、単に「要求されたコードを書く（シングルタスクの完了）」ことだけに集中せず、その変更がシステム全体（バックエンド、データベース、Git構成、実行環境）に与える影響を多角的に検証し、副作用（バグ・クラッシュ）をゼロにする責任がある。
すべての開発・デプロイ作業において、以下の**「4つの検証レイヤー」**に基づく自己検証（セルフチェック）を実行すること。

### 1. データベース（DB）移行・スキーマ変更レイヤー
変更にDBの構造変更や移行が含まれる場合、以下を必ず検証する。
- **データ整合性:** 既存のデータが新しいスキーマで破損、または消失（Nullエラーや型不一致など）しないか。
- **ロールバックの担保:** 移行（Up）が失敗した場合、または本番で問題が発生した場合に、安全かつ即座に元の状態に戻す手順（Down）やフォールバックが確立されているか。
- **N+1問題・パフォーマンス:** インデックスの再構築や、クエリの書き換えによってバックエンドの処理速度が極端に低下しないか。

### 2. コード品質・構文・ランタイムエラー防止レイヤー
- **静的解析・構文チェック:** 記述したコードにシンタックスエラー（構文エラー）、タイポ、インポート漏れ（未定義の変数・関数の参照）がないか。
- **依存関係の衝突:** 新しいライブラリやパッケージを追加・更新した場合、既存の依存関係（package.json 等）と衝突しないか、また実行環境（AWSコンテナ等）でネイティブバイナリの互換性エラーを起こさないか。
- **エラーハンドリング:** バックエンド処理で例外が発生した場合に、システム全体がクラッシュせず、適切にキャッチしてエラーレスポンスとログを吐き出す構造になっているか。

### 3. Gitプッシュ・コンフリクト回避レイヤー
- **影響範囲の最小化:** 変更対象外のファイルや、不要な設定ファイル、機密情報（環境変数など）を誤ってコミットやステージングに含めていないか。
- **コンフリクト回避:** ターゲットブランチへマージ・デプロイする際、コンフリクトが発生する可能性のある既存コード（特に共通関数や共通モジュール）はないか。

### 4. 周辺機能への影響（サイドエフェクト）検証レイヤー
- **結合バグの予測:** 今回修正した項目（API、関数、コンポーネント）を呼び出している「他の機能」や「フロントエンド/バックエンドの通信」が壊れないか。
- **状態管理の整合性:** グローバルな状態（State）やセッション、キャッシュの仕様変更により、別画面や別処理で予期せぬ挙動が発生しないか。

## 🚨 必須ルール9: ローカル同期除外とデプロイの仕様

コンテキストの切り替えや会話の圧縮に伴うAIの忘却を防ぐため、以下の最新の開発ルールと認証仕様を厳守してください。

### 1. デスクトップのローカルフォルダへの同期【除外】
- **開発ワークスペースでのコード修正を、デスクトップのローカルプレビューフォルダ (`C:\Users\one\Desktop\RetailMedia_System`) へ上書き同期するルールは廃止（ルールから除外）** されました。
- 今後は、ローカルプレビューフォルダへのコピー・同期処理は行わないでください。

### 2. Gitプッシュおよび本番環境へのデプロイ【必須】
- すべてのコード修正完了後は、必ずGitへのコミット＆プッシュ（`git push origin main`）を行い、本番環境（AWS App Runner）へのデプロイを走らせて正常に完了したことを報告してください。

### 3. 利用サービス記憶と自動補完 (シームレスUX) 仕様
- 各ログイン画面に「ログイン状態と遷移を維持する」チェックボックス (`#remember-me` / `#remember`) を配置し、チェックONの状態でログイン成功時のみ `last_active_service`（遷移先URL）、`last_email`（メールアドレス）、`last_password`（パスワード）を顧客のスマホ（LocalStorage）に保存します。
- ログアウト時でも、上記記憶データは削除せず維持し、次回起動時（`index.html` アクセス時）に自動的にそのログイン画面へリダイレクトされ、認証情報が自動入力された状態からスムーズに復帰（再ログイン）できる設計となっています。

## 🚨 必須ルール10: CORSブロック発生時の対応ルール (CORS Block Resolution)

CORSブロックの問題が発生した場合は、CORSポリシーで制限をかけるのではなく、速やかに以下の方針を適用して接続を確保してください。

1. **アクセス許可元（Origin）に `*`（すべてのドメイン）を指定する**
   CORSブロックの解消を最優先とするため、レスポンスヘッダーの `Access-Control-Allow-Origin` に `*` を指定して、すべてのドメインからのアクセスを許可する設定に変更してください。
2. **credentials が不要な場合、またはセッションをトークンなどで処理する場合**
   `Access-Control-Allow-Origin: *` と `Access-Control-Allow-Credentials: true` の同時返却はブラウザ仕様でエラーとなるため、CORSエラーを解決するために `credentials` の受け渡し（クッキー等）を解除するか、他の手段に代替した上で `*` による全許可を優先してください。

## 🚨 必須ルール11: AI開発メタ・ルール (AI Rules & Persistence Policy)

今後、ユーザーから**「ルール化して」という指示や要求があった場合は、AIは別の手段や返答だけで済ませず、必ず自動的かつ最優先で本ドキュメント（`BACKEND_RULES.md`）にその内容を明文化して書き加え、永続化すること**を絶対ルールとします。

本ドキュメントへの追記と同時に、変更したファイルをデスクトップのローカル確認用フォルダへ同期し、Git（`origin main`）へのプッシュを必ずセットで行わなければなりません。

---

## 🚨 必須ルール12: エラー報告時のデバッグログ追加ルール (Debug Logging Policy on Error Report)

今後、ユーザーから「エラー」の報告や指摘があった場合は、AIは推測や修正コードの提示だけで終わらせず、**必ずエラーが発生している該当箇所、関連するバックエンドサービス、およびHTML等のフロントエンドの処理（特にイベントハンドラやAPI呼出部）に対して、F12デベロッパーツールで追跡可能なデバッグ用ログ（`console.log` 等）を最先頭および主要な分岐点へ直ちに追加すること**を絶対ルールとします。

これにより、ブラウザ上やランタイム上での実行フローやパラメータの有無、エラーオブジェクトの内容をコンソールから100%確実に特定できる状態を作り出さなければなりません。
